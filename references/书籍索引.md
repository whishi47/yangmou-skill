# 阳谋书单与灌书规范

本 skill 的知识底座以"结构化案例"为核心，而非堆砌原书。任何一本书，都要拆成 `cases.json` 里的一条或多条案例，再进入检索与多层分析。

## 二、灌书拆解规范（新增案例必填字段）

每读一本书/一个真实商业案例，按以下结构写入 `references/cases.json` 的 `cases` 数组：

```json
{
  "id": "tuienling",
  "name": "推恩令",
  "era": "西汉",
  "type": "历史",
  "pillars": ["规则"],
  "summary": "允许诸侯封地诸子均分，表面推恩实则瓦解诸侯。",
  "background": "汉初藩王势大，硬削藩引发七国之乱。",
  "actors": ["汉武帝", "主父偃", "诸侯王及其子弟"],
  "lock_mechanism": "拒绝=与亲生儿子为敌且失道义；接受=封地越分越小，几代后自愈。",
  "open_move": "公开下诏'推恩'，把集权包装成皇家恩典。",
  "outcome": "不流血削弱诸侯，中央集权稳固。",
  "sales_transfer": "把'只对大客户有利'的政策，设计成让小客户也觉得受益，自然扩大覆盖。",
  "tags": ["制度设计", "分化", "分封", "集权", "温水煮青蛙"]
}
```

字段说明：
- `pillars`：从 `规则 / 人性 / 大势` 中选，可多选。
- `lock_mechanism`：**最关键**字段，描述对手为何"无解"。
- `transfer_models`：通用思维模型，描述该阳谋可迁移的"势"逻辑（跨领域适用，不限于销售）。
- `tags`：检索关键词，尽量覆盖同义词（如"分化/拆分/诸子均分"）。

## 三、扩展建议

- 持续往 `cases.json` 增加真实商业阳谋（你所在行业的案例最佳）。
- 行业越垂直，检索命中越准。各领域均可补充：SaaS 免费转付费、会员锁客、公开报价锚定、渠道返利设计、跨部门推项目、长期持有说服、产品定价重构等（七领域皆可）。
- 若灌入整本书，先用本规范拆出 5~15 条高密度案例即可，不必逐句搬运。
